Section 3 — Game Overview Summary

★ What is the game? (1-2 sentences)

SkillForge SAD is a browser-based, scenario-driven gamified platform where students take on the role of a junior system analyst hired by fictional companies. Each session presents a realistic client brief full of contradictions and ambiguity — students must apply SAD tools (interviews, use cases, domain models, UML, ER diagrams) to resolve the brief and "deliver the system" before the deadline runs out.

★ What is the purpose?

To improve student ability to correctly apply requirements elicitation techniques, construct accurate UML use case and domain models, and make structured architecture decisions — targeting a measurable ≥ 20% improvement in practical assignment scores compared to the previous cohort without gamification.

★ What will be taught?

Map to specific SAD topics from the Satzinger textbook.
The game teaches the following specific COM4061 topics (mapped to Satzinger chapters):

• Requirements Elicitation — identifying functional and non-functional requirements from an ambiguous client brief (Satzinger Ch. 4–5)
• Use Case Modeling — writing use case descriptions and constructing use case diagrams with correct actor boundaries, include/extend relationships (Satzinger Ch. 6)
• Domain Modeling — identifying domain classes, attributes, and associations from problem descriptions (Satzinger Ch. 7)
• UML Sequence & Activity Diagrams — modeling system interactions and workflows (Satzinger Ch. 8–9)
• Database Design — translating domain models into ER diagrams and relational schemas (Satzinger Ch. 14)
• System Architecture — choosing between architectural patterns and justifying the decision with trade-off analysis (Satzinger Ch. 17)

★ How will learning be achieved?

Why does playing this game develop the targeted skill?
Each game session presents a Client Case — a realistic brief written with intentional contradictions, missing information, and stakeholder conflicts (e.g. the client says "no login required" but also "each user sees personalised data"). Students must:

1. Read and annotate the brief to extract functional requirements
2. Identify missing information and write elicitation questions
3. Construct the use case diagram for the described system
4. Build the domain model (classes, attributes, associations)
5. Submit their model — the game auto-validates structural correctness and awards XP

The contradiction-resolution mechanic forces students to actively think like analysts, not passive receivers. Immediate auto-feedback after each submission shows exactly what is correct or missing — creating a tight feedback loop that lectures cannot provide. Team Boss Challenges require groups to agree on a single model, developing collaborative analysis skills directly applicable to real project work.

★ How will outcomes be measured? (Kirkpatrick L1–L3 minimum)

L1 — Reaction: 5-question in-app survey completed immediately after each session (scale 1–5). Target: average ≥ 4.0/5.0. Results visible on the Admin Dashboard in real time.

L2 — Learning: Baseline quiz at Week 1 (before game starts) vs. Week 7 mid-point quiz vs. Week 14 final quiz. All three quizzes test the same core skills: requirement identification, use case construction, domain modeling. Target: ≥ 20% score improvement from baseline to Week 14.

L3 — Behaviour: Instructor compares COM4061 practical assignment rubric scores (use case accuracy, domain model completeness) between the gamified cohort and the previous year's cohort. Transfer is confirmed if gamified cohort scores ≥ 15% higher on use case and domain model criteria.

L4 — Results (tracked but not primary): End-of-semester exam score comparison vs. previous cohort. Reported in post-semester retrospective.

Section 5 — Game Mechanics

★ Primary Mechanics

1. Client Case Scenario (Core Mechanic)
Each session presents a realistic fictional client brief (e.g. "We need a hospital appointment booking system — every patient must stay anonymous but also have a personal history"). Students annotate the brief, extract FRs/NFRs, and build the required model. The contradiction-resolution mechanic forces active thinking rather than passive reading — this is why it develops analyst skills, not just engagement.

2. Model Validator (Auto-feedback Engine)
After each submission, the system checks the student's use case diagram or domain model against a rubric (correct actor boundaries, correct include/extend usage, correct class associations). Immediate, specific feedback is given — e.g. "Actor boundary missing between User and Payment System." This tight feedback loop is the core learning mechanism: students correct errors in real time, not weeks later on a returned assignment.

3. Team Boss Challenge (Collaborative Pressure)
Once per week, teams of 4 receive a larger, multi-part Client Case. All members must individually submit a section of the model (e.g. one does use cases, one does domain model, one does ER diagram, one does architecture choice). The team's combined submission is validated as a whole. This replicates real-world SAD team dynamics and forces communication about modeling decisions.

4. Adaptive Difficulty Engine
After 3 consecutive wrong submissions, the platform quietly switches the student to a simpler version of the same challenge (e.g. a Client Case with fewer contradictions, or a partially pre-filled diagram). Difficulty restores automatically when the student recovers. No other student sees this switch — it avoids public shame while preventing discouragement.

5. Leaderboard + XP + Badges
Weekly-reset individual leaderboard, persistent team leaderboard, and per-topic sub-leaderboards. XP and badges drive motivation for Achiever and Player types. Weekly reset prevents permanent demotivation for lower-ranked students — every Monday is a new start.

6. Peer Review Round
Every other session, students review one anonymised peer submission using a visible 4-criterion rubric (actor completeness, relationship correctness, notation accuracy, naming clarity). Reviewing a peer's model requires the student to apply the same analytical skills as building their own — this targets L4 Analyse and L5 Evaluate in Bloom's taxonomy.
